# SECTORCALC SUPERV4 FORMULA GENERATION PROMPT V5.3.1

V5.3.1 is a backward-compatible hardening release of V5.3. It preserves the approved schema top-level key set and the SuperV4 UniversalIndustrialDecisionForm target while adding mandatory formula/form integration safeguards for redaction type consistency, executable state-machine binding, approved schema fetch, private formula registry boundaries, proof-pack module separation, internal checker trace restriction, golden hash storage, root-only guard verification, and final integration reporting.

## ROLE
Act as a Senior Industrial Calculation Architect, Standards-Context Engineer, Unit-Safety Engineer, Measurement-Uncertainty Analyst, Formula-Governance Engineer, Public-Formula-Protection Architect, B2B Decision-Interpretation Designer, Dynamic Form Contract Designer, and Runtime-Safety Reviewer.

## OBJECTIVE
Generate one production-grade SectorCalc SuperV4 calculator JSON schema for a single industrial decision operation.

The generated schema must run without custom per-tool UI code inside the SectorCalc SuperV4 UniversalIndustrialDecisionForm. The schema must define the form contract, unit contract, input reference range contract, validation contract, formula graph contract, output contract, decision interpretation contract, business-impact contract, proof-pack contract, audit contract, UI rendering contract, runtime binding contract, and test plan for one calculator.

The generated calculator must not behave like a basic calculator. It must behave like a paid, upper-segment, industrial decision-support tool that helps operators, engineers, shop-floor technicians, estimators, owners, CFOs, and checkers decide whether to proceed, hold, reprice, reject, repair, inspect, derate, or escalate.

## ABSOLUTE LANGUAGE RULE
All generated public UI text, schema labels, tooltips, input names, output names, warning messages, audit labels, proof-pack text, methodology summaries, test names, descriptions, comments, identifiers, slugs, and metadata must be pure technical English only.

Do not output Turkish, non-English visible text, pseudo-localized text, mixed-language UI strings, locale routes, or locale prefixes.

- executionState: idle, validating, blocked, executing, review, ok, error
- serverResponseState: public-safe response only
- auditSealState: public audit seal summary only
- exportState: PDF, JSON audit, copy summary availability

- The form must not bypass the state machine for profile mode, raw input changes, selected unit changes, evidence changes, normalized preview, client precheck, blocker state, server execution, response consumption, audit seal display, or export availability.
- The state machine must own the execution flow from schema initialization to server response display.
- Client precheck may validate required inputs, selected units, unit conversion entries, physical hard bounds, evidence toggles, schema structure, and semantic blockers.
- Client precheck must not execute formulas.
- Normalized preview may perform declarative unit conversion only.
- Outputs, decision interpretation, hidden risks, FMEA, scenario comparison, proof pack, uncertainty, sensitivity, and audit seal must come only from the server response.
- A state transition that is declared but not reachable from UniversalIndustrialDecisionForm is a failed integration.
- A form that keeps an ad-hoc internal execution state instead of the V5.3.1 machine is a failed integration.

State machine verification must prove:
- all required state domains exist;
- all required transitions are dispatchable;
- profile mode changes are machine-owned;
- unit changes preserve physical quantity;
- normalized preview does not call formula execution;
- server submit transitions through validating, blocked or executing, and response states;
- server blockers are stored in blockerState and validationState;
- audit seal state is populated only from public-safe server response.

## CANONICAL REDACTION STATUS CONTRACT
Create and enforce one canonical redaction status contract across schema-facing runtime contracts, TypeScript contract types, audit seal, proof pack, export, tests, and fixtures.

Canonical allowed values:
- `PUBLIC_SAFE_REDACTED`: public response has been redacted and is safe for browser, PDF, JSON audit, and copy-summary output.
- `INTERNAL_TRACE_RESTRICTED`: internal checker trace exists only in restricted server/admin/test context and must not be included in public response.
- `REDACTION_NOT_REQUIRED`: no protected formula, restricted table, proprietary coefficient, or internal trace was present in the response path.
- `REDACTION_FAILED_BLOCKED`: redaction failed and the public response must be blocked.

Rules:
- `ExecuteResponse.redaction_status` and `AuditSeal.redaction_status` must use the same exported type.
- Do not duplicate incompatible literal unions.
- Missing redaction status must fail runtime contract validation.
- Redaction failure must block public response and export.
- Public proof pack, JSON audit export, PDF export, and copy summary must share the same public-safe redaction policy.
- Internal checker trace must use restricted status and must never be transmitted to the browser.

In SCHEMA_GENERATION_MODE, represent this requirement inside `form_runtime_binding.execute_response_contract`, `audit_trail_contract`, `proof_pack`, `export_contract`, `reference_code`, and `test_plan`.
In FORMULA_FORM_INTEGRATION_MODE, implement it as a single source-of-truth TypeScript type and enforce it with tests.
